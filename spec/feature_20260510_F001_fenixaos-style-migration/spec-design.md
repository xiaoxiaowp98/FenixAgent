# Feature: 20260510_F001 - fenixaos-style-migration

## 需求背景

`mothership` 当前前端仍处于非常早期的控制台原型阶段，页面结构简单、样式以内联为主，尚未形成稳定的设计语言。与此同时，`FenixAOS` 已经沉淀出一套比较完整的深色控制台风格：深色工作区、半透明卡片、低饱和背景、高亮功能色、统一的状态表达与导航节奏。

这次重构并不是在旧 UI 上小修小补，而是一次没有历史包袱的重写。目标不是逐页复刻 `FenixAOS`，而是在 `mothership` 中尽早确定一套稳定的视觉系统，让后续页面开发都围绕同一套风格生长，避免先随手堆页面、后期再整体返工。

用户已明确本 feature 的边界：

- 保留 `FenixAOS` 的深色控制台气质
- 允许重新设计布局、比例和细节
- 本次只先输出视觉系统规范，不提前锁死全部页面信息架构

## 目标

- 为 `mothership` 定义一套可复用的深色控制台视觉系统，而不是零散页面样式
- 提炼 `FenixAOS` 中值得继承的设计语言，并转换为更适合 `mothership` 重构工程的规范
- 明确主题变量、颜色语义、间距层级、卡片体系、表单体系、状态表达和壳层原则
- 为后续 Dashboard、Environment、Session、Plugin、Config 等页面提供统一视觉基线
- 降低未来页面扩展和组件重构成本，避免“页面先长歪，再回头统一皮肤”

## 方案设计

### 推荐策略

推荐采用“**先设计系统，后页面实现**”的路径，而不是先挑一个页面做得漂亮再向外复制。

原因如下：

1. `mothership` 仍处于重构早期，页面数量还不多，此时最适合先统一 token 和组件语义，成本最低。
2. `FenixAOS` 真正可迁移的价值在于整体风格语言，而不是某一个页面的具体排版。
3. 如果先做页面、后补规范，后续 Dashboard、会话、环境、配置页很容易出现多个“相似但不一致”的深色风格分支。

因此，本 feature 不直接定义完整页面结构，而是先产出一套可以约束后续实现的视觉系统规范，并要求新页面只从这套规范中取值。

### 方案对比

#### 方案 A：直接复刻 `FenixAOS` 页面与样式

优点：

- 起步快，短期能快速得到“像 FenixAOS”的结果
- 视觉风险低，容易对齐已有参考

缺点：

- 容易把 `FenixAOS` 的页面结构、信息密度和业务痕迹一并带入
- `mothership` 的域模型和交互路径不同，后续会出现“视觉像，结构不合适”的问题
- 很难沉淀成一套干净的系统规范

结论：不推荐。

#### 方案 B：提炼设计语言，重新定义 `mothership` 的视觉系统

优点：

- 能保留深色控制台气质，同时避免复制旧页面包袱
- 适合重构阶段，便于后续扩展更多页面与组件
- 更容易形成统一 token、统一组件和统一壳层

缺点：

- 前期需要先做一轮抽象，不能马上开始“随便画页面”
- 需要团队在一开始接受规范优先，而不是局部最优

结论：推荐采用。

#### 方案 C：只迁移颜色和卡片皮肤，其他等页面开发时再决定

优点：

- 前期投入最小
- 适合只想先把页面从“难看”变成“没那么难看”

缺点：

- 实际上只是换皮，不是设计系统
- 导航、布局、信息密度、状态组件依然会分裂
- 很容易在几周后再次进入整体整理期

结论：不推荐。

### 视觉定位

`mothership` 的目标风格定义为：

- 深色控制台
- 偏工程控制面板，而不是营销站点
- 视觉克制，但允许少量高亮色承担“功能分区”和“状态反馈”
- 以高可读性和高信息密度为第一原则
- 页面氛围接近“运行中的系统工作台”，而不是“通用后台模板”

从 `FenixAOS` 中继承的核心特征：

- 深色背景与浅色文本的高对比基础
- 卡片、面板、列表的分层依赖亮度差与边框，而不是大面积纯色块
- 使用蓝、绿、黄、橙等功能色表达状态、模块与重点
- 主界面以侧边栏 + 工作区为基础骨架
- 监控、构建、模型等功能页共享同一套卡片与标题节奏

本次不直接继承的部分：

- `FenixAOS` 中部分硬编码颜色类名
- 某些页面的业务专属布局比例
- 不够系统化的“页面内自行决定颜色和容器层级”的写法

### 设计系统分层

视觉系统按四层建立：

1. `Design Tokens`
   颜色、圆角、间距、阴影、边框透明度、排版层级、动效时长
2. `Semantic Tokens`
   如 `surface/base`、`surface/panel`、`surface/elevated`、`text/primary`、`status/success`
3. `UI Primitives`
   Button、Card、Badge、Input、Tabs、SidebarItem、PanelHeader、SectionBlock
4. `Page Shell Patterns`
   控制台壳层、双栏/三栏工作区、列表页、详情页、监控页、空状态页

核心约束：

- 页面不得直接发明颜色值，必须优先使用 token
- 页面不得各自定义“差不多”的卡片容器，必须复用统一 surface 层级
- 状态颜色必须来自统一状态语义，不允许每页自己定义 green-500 / blue-500 之类的组合

### 主题与颜色规范

推荐以 `FenixAOS` 的深色主题为原型，但在 `mothership` 中改造成更明确的语义 token。

#### 基础色角色

- `bg.canvas`：应用最外层背景，承担整页氛围
- `bg.surface`：主工作区基础面
- `bg.panel`：卡片、侧栏、浮层的默认容器底色
- `bg.panel-muted`：弱层级容器，如筛选区、空状态提示区
- `bg.elevated`：更高层级浮层，如弹窗、悬浮面板
- `border.default`：默认边框
- `border.strong`：强调边界
- `text.primary`：主文案
- `text.secondary`：说明文案
- `text.muted`：弱提示
- `text.inverse`：反色文案

#### 状态语义色

- `status.info`：蓝色，用于运行中、连接中、引导动作
- `status.success`：绿色，用于成功、在线、通过
- `status.warning`：黄色或橙色，用于处理中、提醒、待确认
- `status.danger`：红色，用于错误、删除、异常
- `status.neutral`：灰蓝色，用于空闲、离线、未配置

#### 模块功能色

允许保留轻量模块功能色，但只用于图标底片、统计卡片点缀和导航激活提示，不允许泛滥到整块页面背景。

建议初始分配：

- Dashboard / Overview：蓝
- Environment / Runtime：青蓝
- Session / Chat：紫蓝
- Plugin / Capability：绿色
- Config / Settings：橙黄
- Danger / Destructive：红

#### 使用原则

- 背景层级主要通过明度差和透明度区分，不靠纯黑纯灰硬切
- 高亮色只做 10% 到 15% 的弱背景或图标点缀，不做大面积铺底
- Badge、统计图标容器、左侧导航激活态应共享同一色彩语义

### 排版规范

遵循仓库约束，使用系统字体栈，不引入外部字体资源。

建议层级：

- 页面标题：`28-32px`，半粗到粗，紧凑行高
- 区块标题：`18-20px`
- 卡片标题：`15-16px`
- 正文：`14px`
- 次级说明：`12-13px`
- 数据大数字：`24-32px`

排版原则：

- 标题密度高于通用后台，减少空洞的大段留白
- 说明文字始终控制在次级色，不与主标题抢视觉
- 页头、区块头、卡片头维持统一节奏，不每页单独发明

### 空间与容器规范

推荐以 4px 为基础节拍，优先使用 `4 / 8 / 12 / 16 / 24 / 32` 的层级。

容器规则：

- 页面左右内边距默认 `24-32px`
- 卡片内边距默认 `16-24px`
- 卡片组 gap 默认 `16px`
- 统计卡片、列表筛选、详情面板统一使用相同圆角体系

圆角建议：

- 小元素：`8px`
- 输入框、按钮、Badge：`8-10px`
- 卡片、面板：`12-16px`
- 大型容器：`16-20px`

### Surface 体系

这是本次规范的核心之一。`mothership` 后续所有页面都应只在有限的 surface 层级中组合，不再自行决定“这一块是深一点还是浅一点”。

建议定义：

- `surface-0`：页面最底层背景
- `surface-1`：默认内容区和大面板
- `surface-2`：卡片、列表项、导航容器
- `surface-3`：浮起的重点区块、悬浮区域、弹窗

每层同时规定：

- 背景色
- 边框透明度
- 阴影强度
- 是否允许 `backdrop-blur`

建议策略：

- 侧边栏、主内容区、右侧详情面板可以共享同一套 surface 语义，只改尺寸和边界，不改设计语言
- 半透明和 blur 只用于“高级容器”，不要把整站都做成玻璃态

### 组件皮肤规范

#### Button

需要至少定义以下语义变体：

- `primary`：主动作，蓝色系
- `secondary`：次动作，低对比面板色
- `outline`：边框动作
- `ghost`：列表或工具栏轻操作
- `danger`：删除、断开、停用

规则：

- 所有按钮 hover 通过亮度和边框变化反馈，不依赖大幅位移动画
- 图标按钮与文本按钮共享尺寸体系
- 危险操作在默认态也应具备足够辨识度

#### Card / Panel

所有业务卡片统一具备：

- 清晰的背景层级
- 柔和边框
- 有限阴影
- 统一圆角和 padding

Card 应支持以下结构插槽：

- `header`
- `title`
- `description`
- `content`
- `footer`
- `action`

#### Input / Select / Textarea

表单控件应继承深色控制台风格，而不是浏览器默认浅色输入框。

规范：

- 默认背景比卡片底色略深或略亮，但必须可区分
- Focus 使用统一 ring 颜色，不允许每个表单项各自定义
- 错误态采用红色边框 + 轻量 ring，不使用过度刺眼的整块红底

#### Badge / Status Pill

Badge 分两类：

- `status badge`：表达运行态、在线态、错误态
- `category badge`：表达 provider、plugin、capability、mode

其中状态类必须完全绑定语义 token，类别类可以使用较弱的彩色底片，但必须保证对比度。

### 导航与壳层原则

本次不锁定最终页面 IA，但先规定壳层方向。

推荐壳层：

- 左侧固定主导航
- 中间主工作区
- 可选右侧详情区或上下分区

导航设计原则：

- 使用深色面板承载，而不是浅底 sidebar
- 当前激活项通过背景、边框或色条明确高亮
- 图标颜色可轻量承担模块分类语义
- 分组与折叠应服务于信息架构，不要为了“像 FenixAOS”而机械复制

壳层必须优先支持以下几类页面：

- Dashboard 概览页
- Environment / Instance 列表页
- Session / Relay / Chat 工作页
- Plugin / Config 列表 + 详情页
- 监控/状态页

### 页面模式规范

即使本次不展开详细页面设计，也要先约束常见页面模式。

#### 概览页

- 顶部页头：标题 + 简介 + 主动作
- 首屏指标卡：2 到 4 个一组
- 下方是列表或重点模块入口

#### 列表页

- 顶部页头
- 筛选/搜索工具条
- 主列表卡片或表格容器
- 批量操作区与空状态区样式统一

#### 工作台页

- 强调面板分栏关系
- 左中右栏之间靠 surface 与 border 分层，而不是大色块切割
- 面板头部样式统一

#### 详情页

- 基本信息卡
- 状态卡
- 分段内容区
- 相关操作区

### 动效与反馈

建议整体动效克制，重点放在状态可读性而不是炫技。

建议：

- Hover、focus、panel expansion 使用 `120ms - 180ms`
- 弹窗、sheet、popover 使用 `180ms - 220ms`
- 避免过多 scale、bounce、浮夸阴影
- Loading、empty、error、success 要有统一组件表达

### 与当前 `mothership` 的衔接方式

当前 `mothership/apps/web` 仍是最小化 React + Vite 工程，因此视觉系统落地时建议按以下顺序进行：

1. 先引入全局样式入口与 token 体系
2. 再建立最小的 primitives 层
3. 再建立 console shell
4. 最后让 Dashboard 成为第一批规范消费页

这样能避免在早期就出现大量内联样式和一次性页面结构。

建议初始目录方向：

- `src/styles/`：主题 token、全局基线、motion
- `src/components/ui/`：button、card、input、badge、tabs、dialog 等 primitives
- `src/components/shell/`：app-shell、sidebar、topbar、panel-header
- `src/components/patterns/`：summary-card、empty-state、status-row、filter-bar

## 实现要点

- 不要直接复制 `FenixAOS` 的硬编码 class 作为长期方案，应先翻译为 `mothership` 自己的 token
- 第一阶段先把当前 dashboard 的内联样式替换为全局样式与 primitives，作为系统试点
- 页面开发时新增组件优先进入 `ui/` 或 `patterns/`，避免直接在页面里写一次性样式
- 由于当前 web 工程依赖很少，视觉系统初期应保持轻量，不强依赖完整第三方组件体系
- 即使未来引入组件库，也要由 `mothership` 的语义 token 驱动，不反过来让组件库决定站点风格
