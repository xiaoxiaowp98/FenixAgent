# 系统提示词

系统提示词是塑造 Agent 行为最重要的手段。本章介绍 RCS 中所有能注入 Agent 行为的指令入口，以及如何编写有效的提示词。

## 指令注入的三层结构

Agent 接收到的指令来自三个层级，优先级从低到高：

```
项目指令文件（CLAUDE.md）    ← 全局性的项目约定，所有 Agent 共享
    ↓
Agent 系统提示词（prompt）    ← 角色专属的行为定义
    ↓
Skill 指令（SKILL.md）       ← 按需加载的能力指令
```

## 项目指令文件

### CLAUDE.md

`CLAUDE.md` 是项目级全局指令文件，放在 workspace 根目录。Agent 运行时会自动读取此文件，作为所有对话的基础上下文。

```
my-project/
├── CLAUDE.md          ← Agent 自动加载此文件
├── src/
├── package.json
└── ...
```

**适合放在 CLAUDE.md 中的内容**：

- 项目技术栈和架构概述（"这是一个 Next.js 14 + PostgreSQL 项目"）
- 代码规范约定（"使用函数式组件，禁止 class 组件"）
- 目录结构说明（"API 路由在 src/app/api/，页面在 src/app/ 下"）
- 测试约定（"测试文件与源文件同目录，命名为 *.test.ts"）
- 安全约束（"永远不要暴露数据库连接字符串"）

**示例**：

```markdown
# 项目指南

## 技术栈
- 后端：Elysia + Bun
- 数据库：PostgreSQL + Drizzle ORM
- 测试：Bun test

## 代码规范
- 使用 TypeScript strict 模式
- 文件命名：kebab-case
- 组件命名：PascalCase
- 禁止使用 as any

## 测试要求
- 新功能必须包含测试
- 测试文件放在 src/__tests__/ 目录下

## 注意事项
- 不要修改 .env 文件
- 数据库变更必须通过 drizzle-kit generate 生成迁移
```

### 子目录指令文件

`CLAUDE.md` 也可以放在子目录中，当 Agent 操作该目录下的文件时，会读取对应层级的指令文件：

```
my-project/
├── CLAUDE.md              ← 全局指令
├── src/
│   ├── CLAUDE.md          ← src/ 目录下的局部指令
│   └── api/
│       └── CLAUDE.md      ← src/api/ 下的更细粒度指令
```

### 指令文件命名

不同 Agent 运行时识别的指令文件名可能不同。RCS 当前主要使用 opencode 运行时，识别 `CLAUDE.md`。如果你的 Agent 兼容 Claude Code，也是 `CLAUDE.md`。

| 运行时 | 指令文件名 |
|--------|-----------|
| opencode | `CLAUDE.md` |
| Claude Code | `CLAUDE.md` |

## Agent 系统提示词

在 RCS 控制台的 Agent 配置中，`prompt` 字段是 Agent 的核心系统提示词。它定义了 Agent 的角色、能力和行为边界。

### prompt 的定位

`prompt` 是 Agent 专属的指令，比项目指令文件更具体。每个 Agent 可以有不同的 prompt，定义各自的角色。

### 编写有效的 prompt

一个结构良好的 prompt 包含四个部分：

**1. 角色定义** — 告诉 Agent 它是谁

```markdown
你是一个资深的全栈工程师，擅长 TypeScript、React 和 PostgreSQL。
你的职责是理解用户需求并编写高质量的代码。
```

**2. 工具使用引导** — 告诉 Agent 应该怎么工作

```markdown
## 工作流程
1. 先用 grep 搜索相关代码，理解现有实现
2. 制定修改方案，说明要改哪些文件、为什么
3. 编写代码，遵循项目现有风格
4. 运行测试确认没有破坏现有功能
```

**3. 输出约定** — 告诉 Agent 输出什么格式

```markdown
## 输出规范
- 代码修改时，先说明改了什么、为什么改
- 提供完整的文件内容，不要省略
- 如果涉及数据库变更，同时生成迁移文件
```

**4. 边界约束** — 告诉 Agent 不能做什么

```markdown
## 约束
- 不要删除或修改测试文件，除非用户明确要求
- 不要修改 package.json 中的版本号
- 遇到不确定的问题，先问用户再行动
```

### Agent 角色设计示例

你可以根据团队需要自定义任意角色的 Agent。以下是常见的角色设计方向：

| 角色方向 | 典型职责 | prompt 重点 |
|----------|---------|------------|
| 代码编写 | 编写和修改代码 | 工具使用引导、代码风格约束、测试要求 |
| 任务规划 | 分析需求，制定执行计划 | 分析框架、输出结构化计划、不直接修改代码 |
| 搜索探索 | 搜索和理解代码库 | 搜索策略、结果整理、只读权限 |
| 标题生成 | 为会话生成简短标题 | 简洁输出、语言风格约束 |
| 内容摘要 | 总结对话内容 | 信息提取、格式化输出 |

## 提示词编写最佳实践

### 要做的事

- **具体而非抽象** — "使用 React 函数式组件"比"使用最佳实践"更有效
- **给出示例** — "输出格式如下：\n```json\n{ \"name\": \"xxx\" }\n```"比"输出 JSON 格式"更可靠
- **明确优先级** — "优先使用项目已有的工具函数，避免重复实现"
- **包含错误处理指导** — "如果 API 返回 429，等待 5 秒后重试"

### 避免的事

- **不要过度约束** — 太多限制会让 Agent 变得刻板，失去灵活性
- **避免模糊指令** — "做得好一点"没有可操作性
- **不要在 prompt 中放敏感信息** — prompt 不是加密存储的，API Key 等应通过环境变量注入
- **避免自相矛盾** — 不要同时说"始终使用 TypeScript"和"可以用 JavaScript"

### prompt 与 Skill 的分工

| 场景 | 放在 prompt 中 | 放在 Skill 中 |
|------|---------------|--------------|
| Agent 的核心身份和行为 | ✅ | ❌ |
| 项目特定的编码规范 | ❌ | ❌（放 CLAUDE.md） |
| 可复用的工作流程（如"代码审查"、"部署"） | ❌ | ✅ |
| 按需加载的能力（不是每次对话都需要） | ❌ | ✅ |
